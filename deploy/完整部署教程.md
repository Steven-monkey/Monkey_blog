# 个人博客完整部署教程 —— 从零到 HTTPS

本篇记录将个人博客部署到阿里云 Ubuntu 服务器的全过程，包括所有遇到的坑和解决方案。

---

## 环境信息

| 项目 | 信息 |
|------|------|
| 服务器 | 阿里云 ECS |
| 公网 IP | 120.78.121.40 |
| 系统 | Ubuntu 24.04.4 LTS |
| 域名 | www.moneky.top |
| 技术栈 | Flask + SQLite + Docker + Nginx |

## 第 1 步：SSH 连接服务器

本地终端执行：

```bash
ssh root@120.78.121.40
```

---

## 第 2 步：安装 Docker

### 2.1 安装前置依赖

```bash
sudo apt update && sudo apt install -y ca-certificates curl
```

### 2.2 添加 Docker 官方 GPG 密钥

```bash
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc
```

### 2.3 添加 Docker 软件源

```bash
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
```

### 2.4 安装 Docker 和 Docker Compose

```bash
sudo apt update && sudo apt install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin
```

### 2.5 验证

```bash
docker --version && docker compose version
```

## 第 3 步：配置 Docker 镜像加速

> **坑点**：国内无法直接拉取 Docker Hub 镜像，会报 `docker.io/library/python:3.12-slim: not found`。

```bash
sudo mkdir -p /etc/docker
sudo tee /etc/docker/daemon.json << 'EOF'
{
  "registry-mirrors": [
    "https://docker.1ms.run",
    "https://docker.xuanyuan.me"
  ]
}
EOF

sudo systemctl daemon-reload && sudo systemctl restart docker
```

验证加速生效：

```bash
docker info 2>/dev/null | grep -A3 "Registry Mirrors"
```

mkdir -p /www && cd /www

## 第 5 步：创建 .env 配置文件

`.env` 文件包含 API 密钥，不会提交到 Git，需要在服务器手动创建。

```bash
cd /www/blog
cat > .env << 'EOF'

LLM_API_URL=https://api.siliconflow.cn/v1
IMAGE_MODEL=doubao-seedream-4-5-251128
IMAGE_API_KEY=你的生图密钥
IMAGE_API_BASE=https://ark.cn-beijing.volces.com/api/v3
IMAGE_WIDTH=1080
IMAGE_HEIGHT=1920
IMAGE_FALLBACK_WIDTH=1440
IMAGE_FALLBACK_HEIGHT=2560
HTTP_TIMEOUT_SECONDS=45
LLM_HTTP_TIMEOUT_SECONDS=30
IMAGE_HTTP_TIMEOUT_SECONDS=60
DOWNLOAD_HTTP_TIMEOUT_SECONDS=45
HOST=0.0.0.0
PORT=8080
EOF

chmod 600 .env
```

## 第 6 步：修改 Dockerfile 使用国内 pip 源

> **坑点**：docker build 时 `pip install` 下载包也会被墙，BrokenPipe。

编辑 `/www/blog/Dockerfile`，pip 安装加上清华源：

```dockerfile
FROM python:3.12-slim

WORKDIR /app

RUN mkdir -p /app/data

COPY requirements.txt /app/requirements.txt
RUN pip install --no-cache-dir -i https://pypi.tuna.tsinghua.edu.cn/simple -r /app/requirements.txt

COPY blog/ /app/blog/
COPY content/ /app/content/

ENV HOST=0.0.0.0
ENV PORT=8080
ENV SQLITE_PATH=/app/data/blog.db

EXPOSE 8080

CMD ["python", "-m", "blog.main"]
```

## 第 10 步：申请 SSL 证书

### 10.1 先确认 80 端口空闲

```bash
ss -tlnp | grep :80
```

如果 Docker 的 Nginx 在跑，先停掉：

```bash
cd /www/blog && docker compose stop nginx
```

### 10.2 使用 DNS 验证方式申请证书

> **坑点**：standalone 和 webroot 方式均返回 403，原因可能是云服务商 WAF/CDN 拦截了 Let's Encrypt 的验证请求，也可能是网络环境问题。DNS 验证方式最可靠，完全无需 80 端口。

```bash
sudo apt install -y certbot
sudo certbot certonly --manual --preferred-challenges dns -d www.moneky.top
```

执行后会提示你去 DNS 后台添加一条 TXT 记录：

```
_acme-challenge.www.moneky.top → qJhtyiRJsMCWMpYZWGfNkE0OR1bH6eu4Spq-9Yfmx9w
```

**注意**：每次申请时 TXT 值不同，以终端实际显示为准。

去 DNS 后台添加后，等 1-2 分钟让记录生效，然后回车继续。

证书存放位置：
- 公钥：`/etc/letsencrypt/live/www.moneky.top/fullchain.pem`
- 私钥：`/etc/letsencrypt/live/www.moneky.top/privkey.pem`

### 10.3 证书续期

此方式申请的证书**不会自动续期**，90 天后到期。到期前运行同样命令重新申请即可。

## 第 11 步：配置 Nginx HTTPS

### 11.1 更新 deploy/nginx.conf

```bash
cat > /www/blog/deploy/nginx.conf << 'EOF'
upstream flask_web {
    server web:8080;
    keepalive 8;
}

# HTTP → 强制跳转 HTTPS
server {
    listen 80;
    server_name www.moneky.top;
    return 301 https://$host$request_uri;
}

# HTTPS
server {
    listen 443 ssl;
    http2 on;
    server_name www.moneky.top;

    ssl_certificate /etc/ssl/certs/fullchain.pem;
    ssl_certificate_key /etc/ssl/private/privkey.pem;

    client_max_body_size 25m;

    location / {
        proxy_pass http://flask_web;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Connection "";

        # AI 阅读助手 SSE 流式输出必须关闭缓冲
        proxy_buffering off;
        proxy_cache off;
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;
    }
}
EOF
```

### 11.2 更新 docker-compose.yml

加上 443 端口和证书挂载：

```bash
cat > /www/blog/docker-compose.yml << 'EOF'
services:
  web:
    build: .
    expose:
      - "8080"
    env_file:
      - .env
    environment:
      SQLITE_PATH: /app/data/blog.db
    volumes:
      - sqlite-data:/app/data

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./deploy/nginx.conf:/etc/nginx/conf.d/default.conf:ro
      - /etc/letsencrypt/live/www.moneky.top/fullchain.pem:/etc/ssl/certs/fullchain.pem:ro
      - /etc/letsencrypt/live/www.moneky.top/privkey.pem:/etc/ssl/private/privkey.pem:ro
    depends_on:
      - web

volumes:
  sqlite-data:
EOF
```

### 11.3 重建并启动

```bash
cd /www/blog && docker compose up -d --build
```

## 踩坑总结

| 坑 | 原因 | 解决方案 |
|----|------|---------|
| `python:3.12-slim: not found` | Docker Hub 被墙 | 配置国内镜像加速器 |
| `pip install BrokenPipe` | PyPI 下载被墙 | Dockerfile 中 pip 用清华源 |
| `git clone TLS error` | GitHub 被干扰 | `GIT_SSL_NO_VERIFY=1` |
| `certbot standalone 403` | 云服务商可能拦截验证请求 | 改用 DNS 验证方式 |
| `certbot webroot 403` | Nginx 反代干扰验证路径 | 改用 DNS 验证方式 |
| DNS 验证不会自动续期 | manual 模式限制 | 到期前手动重新申请 |

## 第 13 步：后续更新代码

后续博客代码有更新时，在服务器上执行：

```bash
cd /www/blog && git pull && docker compose up -d --build
```

### 13.1 常见问题：Git 合并冲突

> **坑点**：第一次部署时在服务器上手动改过的文件（Dockerfile, deploy/nginx.conf, docker-compose.yml），后续 `git pull` 更新时会产生合并冲突：
> ```
> error: Your local changes to the following files would be overwritten by merge:
>         Dockerfile
>         deploy/nginx.conf
>         docker-compose.yml
> ```
>
> 因为新代码已经包含了这些改动，直接丢弃本地修改即可：
>
> ```bash
> git checkout -- Dockerfile deploy/nginx.conf docker-compose.yml
> git pull
> docker compose up -d --build
> ```

### 13.2 常见问题：更新后 HTTPS 失效

> **坑点**：`git pull` 拉下来的 `docker-compose.yml` 和 `deploy/nginx.conf` 是通用模板（证书路径是注释/占位符），更新后 HTTPS 会失效，浏览器访问 HTTPS 无法连接。
>
> **原因**：服务器上之前手动配置的证书挂载被通用模板覆盖了。
>
> **修复** — 重新写入带证书的配置：
>
> ```bash
> cd /www/blog
>
> # 1. 更新 docker-compose.yml 挂载证书
> cat > docker-compose.yml << 'EOF'
> services:
>   web:
>     build: .
>     expose:
>       - "8080"
>     env_file:
>       - .env
>     environment:
>       SQLITE_PATH: /app/data/blog.db
>     volumes:
>       - sqlite-data:/app/data
>     restart: unless-stopped
>
>   nginx:
>     image: nginx:alpine
>     ports:
>       - "80:80"
>       - "443:443"
>     volumes:
>       - ./deploy/nginx.conf:/etc/nginx/conf.d/default.conf:ro
>       - /etc/letsencrypt/live/www.moneky.top/fullchain.pem:/etc/ssl/certs/fullchain.pem:ro
>       - /etc/letsencrypt/live/www.moneky.top/privkey.pem:/etc/ssl/private/privkey.pem:ro
>     depends_on:
>       - web
>     restart: unless-stopped
>
> volumes:
>   sqlite-data:
> EOF
>
> # 2. 更新 nginx 配置开启 HTTPS
> cat > deploy/nginx.conf << 'NEOF'
> upstream flask_web {
>     server web:8080;
>     keepalive 8;
> }
>
> server {
>     listen 80;
>     server_name www.moneky.top;
>     return 301 https://$host$request_uri;
> }
>
> server {
>     listen 443 ssl;
>     http2 on;
>     server_name www.moneky.top;
>
>     ssl_certificate /etc/ssl/certs/fullchain.pem;
>     ssl_certificate_key /etc/ssl/private/privkey.pem;
>
>     client_max_body_size 25m;
>
>     location / {
>         proxy_pass http://flask_web;
>         proxy_http_version 1.1;
>         proxy_set_header Host $host;
>         proxy_set_header X-Real-IP $remote_addr;
>         proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
>         proxy_set_header X-Forwarded-Proto $scheme;
>         proxy_set_header Connection "";
>         proxy_buffering off;
>         proxy_cache off;
>         proxy_read_timeout 3600s;
>         proxy_send_timeout 3600s;
>     }
> }
> NEOF
>
> # 3. 重建启动
> docker compose up -d --build
> ```
>
> **小技巧**：如果要避免每次更新都被覆盖，可以在 GitHub 仓库的 `docker-compose.yml` 中直接把证书挂载取消注释并填好路径。但别忘了你的域名和别人不同，所以这只能在自己的 fork 里做。

## 踩坑总结

| 坑 | 原因 | 解决方案 |
|----|------|---------|
| `python:3.12-slim: not found` | Docker Hub 被墙 | 配置国内镜像加速器 |
| `pip install BrokenPipe` | PyPI 下载被墙 | Dockerfile 中 pip 用清华源 |
| `git clone TLS error` | GitHub 被干扰 | `GIT_SSL_NO_VERIFY=1` |
| `certbot standalone 403` | 云服务商可能拦截验证请求 | 改用 DNS 验证方式 |
| `certbot webroot 403` | Nginx 反代干扰验证路径 | 改用 DNS 验证方式 |
| DNS 验证不会自动续期 | manual 模式限制 | 到期前手动重新申请 |
| `git pull` 合并冲突 | 服务器手动改过的文件被新代码覆盖 | `git checkout -- 文件名` 丢弃本地改动 |
| 更新后 HTTPS 失效 | git pull 覆盖了证书挂载配置 | 重新写入 docker-compose.yml + nginx.conf 的证书部分 |

## 日常运维命令

```bash
cd /www/blog

# 查看运行状态
docker compose ps

# 查看日志
docker compose logs -f --tail=100

# 更新代码（注意：如果 docker-compose.yml 或 nginx.conf 有冲突，先 git checkout）
git pull
git checkout -- Dockerfile deploy/nginx.conf docker-compose.yml 2>/dev/null
git pull
docker compose up -d --build

# 更新后如果 HTTPS 失效，重新写入证书配置（参考第 13.2 节）

# 重启单个服务
docker compose restart web     # 改了 Markdown 后重启 web 即可
docker compose restart nginx   # 改了 nginx 配置后
```
